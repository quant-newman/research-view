## G. 持仓 / 自选层(核心,新增维度贯穿全局)

### G0. 为什么必须有
看板此前把175只票一视同仁地展示。但"决策雷达"最该回答的是"**跟我有钱在里面的票,今天发生了什么**"。持仓层让全站从"泛泛的产业链展示"升级为"跟我的钱直接相关的雷达"。核心动作:**我的持仓/自选票,在每一个模块里优先高亮、优先排序、优先推送。**

### G1. 数据模型
两个层级(都由使用者维护,不依赖券商API):
```
持仓 (holdings):   code, name, 状态=持有, [可选]成本价, [可选]仓位比例/金额
自选 (watchlist):  code, name, 状态=关注, 备注(如"等回调"/"跟踪扣非兑现")
```
- **持仓和自选的票不必在175只池内**(你可能持有池外的票,如赛腾股份这类)。池外持仓票也要能纳入监控,单独标注"⚠池外"。
- **隐私处理(重要):** 真实持仓金额是敏感数据。设计成**分级**:
  - 最小可用:只存 `code + 状态`(持有/关注),看板就能做"优先高亮" —— **不含金额也完全工作**。
  - 增强(可选):使用者自愿填成本价/仓位,用于盈亏染色。**金额类数据只存本地SQLite,不进任何会同步/推送的通道,不明文出现在生成的静态页里**(静态页只输出"持有/关注"标记,不输出金额)。
  - Claude Code 默认实现最小可用版,金额字段留接口但默认空。

### G2. 对其他模块的改动(持仓层的价值全在这里)
- **每日报告**:
  - 新增最优先板块 **"▎我的持仓/自选动态"**,置于"今日主线"之后、"今天只看3件事"之前。只讲你持仓/自选票今天的异动+相关新闻。**你的票有事,永远第一时间、最高优先级出现。**
  - top3 里若命中你的持仓/自选,加持仓标记(◆持仓 / ◇自选)并排在前。
- **新闻/事件页**:
  - 筛选栏加一个开关 **"[仅看我的票]"**,一键过滤到只剩持仓/自选相关。
  - 事件流里持仓/自选票的条目加左边框高亮(◆持仓琥珀 / ◇自选蓝),即使不筛选也一眼可见。
- **热力图**:
  - 持仓/自选的票在散点图上加描边/加粗,一眼看到"我的票落在哪个象限"。
  - 监控表加一列"持仓"标记,可按"仅我的票"筛选。
- **推送(见H)**:持仓/自选票的异动是**最高优先级推送**。

### G3. 维护方式
- 一个简单的持仓/自选管理页(增删改),或直接维护一个 `holdings.json` / `watchlist.json`,后端读取。
- 不接券商API(避免敏感授权),纯手动维护。你买卖后自己更新,几秒钟的事。

### G4. 存储
SQLite 加两张表 `holdings` / `watchlist`。金额类字段可空。**`ledger`、`holdings` 金额字段均不进静态导出的公开JSON**(静态页只读"是否持仓/自选"的布尔标记)。

## H. 推送 / 提醒机制

### H0. 定位
静态站不主动打开就不知道盘中变化。推送解决"**重大信号主动找到你**",而不是你去找它。**克制推送,只推真正重要的,避免变成噪音让你麻木。**

### H1. 推送渠道(选一个先做)
- **Telegram Bot**(推荐:免费、API简单、跨境稳定,你在新加坡/台湾无障碍)
- 备选:邮件(SMTP,更简单但你未必及时看)、企业微信/Server酱(国内,但你在境外)
- Claude Code 第一版实现 **Telegram Bot 单渠道**即可。

### H2. 推送什么(分优先级,只推高价值)
- **P0 立即推**:你**持仓/自选票**的重大异动 —— 涨跌停、突发公告(减持/业绩预告/重大合同)、盘中触及你设的价格线。
- **P1 汇总推**:每日报告生成完推一条摘要("盘前报告已生成:今日主线X,你的持仓Y有异动")+ 链接回看板。
- **P2 可选**:你关注节点(非持仓)出现涨停潮/板块异动。
- **绝不推**:普通新闻、非持仓票的日常波动。**宁可少推,不可刷屏。**

### H3. 触发与去重
- 触发在采集管线里判定(每小时那次采集后检查是否命中P0/P1条件)。
- **去重**:同一只票同一事件一天只推一次;盘中价格线触发后当日不再重复推同条件。
- 每条推送带时间戳(UTC+8)+ 回看板链接。

### H4. 价格线提醒(可选增强)
自选表的备注可支持"跌破X提醒/涨破Y提醒",采集时比对触发P0。第一版可省,后续加。

## I. 系统健康监控

### I0. 定位
六模块每天自动跑,**任何一环静默失败都比报错更危险**(你以为数据是新的,其实断了)。监控保证"**挂了你第一时间知道**"。

### I1. 监控什么
每个定时任务(采集/生成/推送)记录:
```
task, status(成功/失败/部分成功), records_count, duration, error_msg, ts_utc8
```
关键检查项:
- **采集成功率**:Tushare/东财/yfinance 各源今天拉到数据没有?条数是否异常偏少(如平时200条今天20条)?
- **DeepSeek 调用**:成功率、超时率、返回是否合法JSON。
- **生成**:盘前/盘后报告是否按时生成。
- **数据新鲜度**:每个模块数据的最后更新时点,超期(如行情超过1小时没更新)报警。

### I2. 失败通知
- 任何 P0 级失败(某源整个挂了/报告没生成)→ **走推送渠道(H)通知你**,与业务推送同一个Telegram Bot,但标 `[系统]` 前缀区分。
- 非致命(某源部分失败但有兜底)→ 记日志,不打扰,看板顶部标一个小黄点提示"部分数据可能不全"。

### I3. 展示
- 看板加一个不起眼的**系统状态页/角标**:各源今日采集状态(绿/黄/红)、各模块数据新鲜度、最近失败日志。
- 顶部状态条:全绿时不显示,有异常时显示黄/红点,点击进状态页。

### I4. 存储
SQLite 加表 `task_log`。轻量,只留最近N天(如30天),自动清理旧日志。

## J. 数据质量校验(第一版最小实现,防脏数据进报告)

> 不单列大模块,但至少加基本校验,因为脏数据比没数据更坏。
- **合理性校验**:单日涨跌幅 >21%(A股涨跌停外)报异常、市值突变>50%报异常、PE为负或极端值标注、停牌/ST票单独标记不参与涨跌染色。
- **DeepSeek输出校验**:返回必须是合法JSON且字段完整,themes必须在固定清单内,tickers必须能在股票池匹配到(匹配不到的标"待核");不合法则该条降级为"仅展示原文不做结构化",不硬塞进报告。
- 校验不通过的数据**标记存疑**而非丢弃,人工可见。

## K. 明确不做(避免过度设计,第一版及可见未来都不做)
- 交易执行/下单(有 Guanlan)
- 回测/策略验证(有 Touchstone/Crucible)
- 多用户/权限/登录(自用工具)
- 组合归因/风险度量VaR(超出"雷达"定位,那是持仓管理系统的事)
- 券商API对接持仓(隐私+复杂度,手动维护足够)
